# v1.2.8 — Batch Input-ID Preflight

This release adds a preflight check for duplicate explicit `input_id` values before batch workers start.

## Why

Batch artifacts are written by stable scrape id. If two CSV rows use the same explicit `input_id`, parallel workers can write into the same folder:

```text
data/scraped/<input_id>/
```

That can corrupt or overwrite row artifacts.

## Behavior

By default, duplicate explicit IDs are made collision-safe in a generated preflight CSV:

```text
SKU1
SKU1__DUP02
SKU1__DUP03
```

The original ID and duplicate metadata are preserved in extra columns:

```text
original_input_id
input_id_duplicate_count
input_id_duplicate_index
input_id_collision_resolved
```

A preflight report is also written next to the output CSV by default:

```text
<output_csv>.preflight.json
```

## Strict mode

Use this option to fail before scraping if duplicate explicit IDs exist:

```bash
--fail-on-duplicate-input-id
```

This does not change scraping behavior. It only protects the batch run from artifact path collisions.

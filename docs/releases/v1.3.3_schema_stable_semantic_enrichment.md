# v1.3.3 — Schema-stable semantic enrichment

This release improves the intelligence density of generated artifacts without changing the artifact output schema.

## What changed

The LLM prompts now ask for richer content inside the existing artifact files and existing schema sections:

```text
product_evidence.json
product_evidence.md
claims.md
vision.md
```

The goal is not to add new files or rename fields. The goal is to make the existing content easier for a downstream product-coding LLM to use precisely.

## Enrichment behavior

Within the existing schema shape, generated rows should now be more explicit about:

```text
attribute name
raw extracted value
normalized value when available
evidence axes
source references
confidence
coding relevance
claim scope / transferability
conflict status
missing reason
downstream decision note
```

## Product-coding focus

The prompts now explicitly prioritize coding-relevant attributes such as:

```text
product name
brand
manufacturer
EAN / GTIN
SKU / MPN
product type / category
franchise / character
age range
piece count
contents / included components
material
dimensions
color / theme
battery or electronic requirements
warnings
package / bundle status
visible toy-specific features
```

## Safety boundaries retained

```text
No search added.
No scraping provider change.
No artifact file rename.
No top-level schema change requested.
No external knowledge.
No guessing.
```

Input context and URL-derived evidence remain provenance signals. They must not be promoted to browser-rendered retailer claims unless supported by rendered text, structured metadata, tables, or visual evidence.

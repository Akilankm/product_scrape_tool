# v1.2.6 — Crawl4AI MarkdownGenerationResult Compatibility

This release fixes a Crawl4AI payload compatibility issue observed during batch execution:

```text
TypeError: object of type 'MarkdownGenerationResult' has no len()
```

## What changed

- Added defensive coercion for Crawl4AI markdown payload variants before logging, scoring, metadata writing, artifact writing, and fail-safe finalization.
- Normalizes payloads such as:
  - plain strings
  - bytes
  - dictionaries
  - Pydantic-like objects
  - `MarkdownGenerationResult`-like objects with `raw_markdown`, `fit_markdown`, `markdown`, `markdown_with_citations`, `text`, or `content` fields
- Ensures batch workers do not fail simply because Crawl4AI returns markdown as an object instead of a string.

## Artifact behavior

The row still follows the same v1.2.5 worker-safe contract:

```text
Every input row must produce either _COMPLETE.json or _FAILED.json.
Every failure still writes a minimal artifact with scrape_result.json, quality_report.json, source_alignment_report.json, product_evidence.json, and vision.md.
```

## Visual evidence contract remains unchanged

Product image evidence remains mandatory for a usable artifact:

```text
final_product_images_available  -> clean image available
screenshot_fallback_only        -> visual rescue, manual review required
image_recovery_failed           -> no usable visual file retained
no_image_candidates             -> no image candidates found in the capture
```

## Validation

The release includes tests covering:

- `MarkdownGenerationResult`-like payload coercion
- merge behavior when non-string markdown leaks into `FullPage`
- failed-artifact finalization
- non-empty `vision.md`
- mandatory image evidence status
